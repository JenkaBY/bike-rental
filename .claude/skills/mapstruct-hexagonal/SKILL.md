# MapStruct – Hexagonal Architecture Patterns

MapStruct 1.6.x is used throughout the service module for type-safe, compile-time mapping between layers. All mappers
are Spring-managed components (generated by the `mapstruct-processor` annotation processor).

## When to Use This Skill

- Creating a new mapper for any layer (web/command, web/query, application, infrastructure/persistence)
- Adding a new shared value-object (VO) mapper in `shared/mapper/`
- Composing multiple mappers with `@Mapper(uses = {...})`
- Handling multi-parameter method signatures
- Mapping to/from JPA entities (audit-field ignoring, `@AfterMapping` for relationships)
- Adding `@Named` qualified conversions
- Injecting Spring beans into an abstract mapper

---

## Mapper Layer Placement

Each module follows a strict placement convention. Use the table below to decide where a new mapper lives:

| Layer           | Package                                       | Responsibility                                                                       |
|-----------------|-----------------------------------------------|--------------------------------------------------------------------------------------|
| Web command     | `<module>/web/command/mapper/`                | HTTP request DTOs → use-case commands; domain → HTTP response DTOs                   |
| Web query       | `<module>/web/query/mapper/`                  | Domain objects / Facade projections → read-only response DTOs                        |
| Application     | `<module>/application/mapper/`                | Use-case commands ↔ domain aggregates; domain → Facade projection DTOs               |
| JPA persistence | `<module>/infrastructure/persistence/mapper/` | Domain aggregates ↔ JPA entities                                                     |
| Module shared   | `<module>/shared/mapper/`                     | Value-object ↔ primitive conversions scoped to one module (e.g. `PhoneNumberMapper`) |
| Global shared   | `shared/mapper/`                              | Universal type conversions reused by all modules (e.g. `MoneyMapper`, `UuidMapper`)  |

## Pattern 1 – Simple Interface Mapper

Use for mappers that do not need any Spring-managed collaborators.

```java
package com.github.jenkaby.bikerental.customer.web.query.mapper;

import com.github.jenkaby.bikerental.customer.CustomerInfo;
import com.github.jenkaby.bikerental.customer.web.query.dto.CustomerSearchResponse;
import org.mapstruct.Mapper;

import java.util.List;

@Mapper
public interface CustomerQueryMapper {

    CustomerSearchResponse toSearchResponse(CustomerInfo customerInfo);

    List<CustomerSearchResponse> toSearchResponses(List<CustomerInfo> customers);
}
```

- List mapping is derived automatically from the single-element method.
- No `@Mapper(uses = ...)` needed when no delegate mappers are required.

## Pattern 2 – Shared Value-Object (VO) Mapper

Create a dedicated `@Mapper` interface for each value object. Place in:

- `shared/mapper/` for globally shared VOs (`Money`, `UUID`, `Instant`, `Duration`, `DiscountPercent`)
- `<module>/shared/mapper/` for module-scoped VOs (`PhoneNumber`, `EmailAddress`, `RentalStatus`)

Rules:

- Implement both directions as `default` methods.
- Guard against `null` on **both** sides.
- Keep each mapper in its own file — one VO per mapper.

```java
// shared/mapper/MoneyMapper.java
@Mapper
public interface MoneyMapper {

    default Money toMoney(BigDecimal amount) {
        return amount != null ? Money.of(amount) : null;
    }

    default BigDecimal toBigDecimal(Money money) {
        return money != null ? money.amount() : BigDecimal.ZERO;
    }
}
```

## Pattern 3 – `@Mapper(uses = {...})` Composition

Reference all delegate mappers that handle type conversions used in the mapping method signatures.
MapStruct picks the right `default` method automatically by matching the source and target types.

```java

@Mapper(uses = {UuidMapper.class, PhoneNumberMapper.class, EmailAddressMapper.class})
public interface CustomerCommandMapper {

    CreateCustomerUseCase.CreateCustomerCommand toCreateCommand(CustomerRequest request);

    CustomerResponse toResponse(Customer customer);
}
```

```java

@Mapper(uses = {MoneyMapper.class, InstantMapper.class, RentalStatusMapper.class,
        DurationMapper.class, RentalOverdueMapper.class, RentalEquipmentWebMapper.class, RentalEquipmentMapper.class})
public interface RentalQueryMapper {

    @Mapping(target = "actualDurationMinutes", source = "rental.actualDuration")
    @Mapping(target = "plannedDurationMinutes", source = "rental.plannedDuration")
    @Mapping(target = "equipmentItems", source = "rental.equipments")
    RentalResponse toResponse(Rental rental);
}
```

## Pattern 4 – `@Mapping` Conventions

### 4a – Ignore Audit Fields (JPA `toEntity` direction)

Audit fields (`createdAt`, `updatedAt`) are managed by JPA auditing and must always be ignored when mapping
**domain → JPA entity**:

```java

@Mapper(uses = {PhoneNumberMapper.class, EmailAddressMapper.class})
public interface CustomerJpaMapper {

    Customer toDomain(CustomerJpaEntity entity);

    @Mapping(target = "createdAt", ignore = true)
    CustomerJpaEntity toEntity(Customer customer);
}
```

```java

@Mapper(uses = {MoneyMapper.class})
public interface TariffJpaMapper {

    Tariff toDomain(TariffJpaEntity entity);

    @Mapping(target = "createdAt", ignore = true)
    @Mapping(target = "updatedAt", ignore = true)
    TariffJpaEntity toEntity(Tariff tariff);
}
```

### 4b – Source Path Disambiguation (multi-parameter methods)

When a mapping method has **more than one parameter**, prefix `source` with the parameter name:

```java

@Mapper(uses = {UuidMapper.class, PhoneNumberMapper.class, EmailAddressMapper.class})
public interface CustomerCommandMapper {

    @Mapping(target = "customerId", source = "customerId")
        // first param (UUID)
    UpdateCustomerUseCase.UpdateCustomerCommand toUpdateCommand(UUID customerId, CustomerRequest request);
}
```

```java

@Mapper
public interface TariffV2CommandMapper {

    @Mapping(target = "id", source = "id")
    @Mapping(target = "params", source = "request.params", qualifiedByName = "paramsToMap")
    UpdateTariffV2UseCase.UpdateTariffV2Command toUpdateCommand(Long id, TariffV2Request request);
}
```

### 4c – Ignore a Generated Field When Mapping `@Mapping(target = "id", ignore = true)`

Use when the target's `id` is generated by the database, not the source:

```java

@Mapper(uses = {MoneyMapper.class}, injectionStrategy = InjectionStrategy.CONSTRUCTOR)
public interface TariffCommandToDomainMapper {

    @Mapping(target = "id", ignore = true)
    Tariff toTariff(CreateTariffCommand command);

    Tariff toTariff(UpdateTariffCommand command);   // id is present in update command
}
```

## Pattern 5 – `@Named` Qualified Conversions

Use `@Named` when **the same source type** needs to map to the same target type in different ways. Reference with
`qualifiedByName` in `@Mapping`.

```java

@Mapper
public interface RentalEquipmentMapper {

    default Long toEquipmentId(RentalEquipment rentalEquipment) {
        return rentalEquipment != null ? rentalEquipment.getEquipmentId() : null;
    }

    List<Long> toEquipmentIds(List<RentalEquipment> rentalEquipment);

    @Named("RentalEquipment.returnedEquipmentIds")
    default List<Long> mapEquipmentsToIds(List<RentalEquipment> equipments) {
        return equipments.stream()
                .filter(re -> re.getStatus() == RentalEquipmentStatus.RETURNED)
                .map(RentalEquipment::getEquipmentId)
                .toList();
    }
}
```

Reference the named method from another mapper:

```java
@Mapper(uses = {RentalEquipmentMapper.class, ...})

public interface RentalEventMapper {

    @Mapping(target = "returnedEquipmentIds", source = "rental.equipments",
            qualifiedByName = "RentalEquipment.returnedEquipmentIds")
    RentalCompleted toRentalCompleted(Rental rental, LocalDateTime returnTime, Money totalCost);
}
```

## Pattern 6 – `@AfterMapping` for JPA Relationship Wiring

Use `@AfterMapping` + `@MappingTarget` to set bidirectional JPA relationships that cannot be derived from field
mapping alone:

```java

@Mapper(uses = {MoneyMapper.class, InstantMapper.class, RentalStatusMapper.class,
        DurationMapper.class, RentalEquipmentJpaMapper.class})
public interface RentalJpaMapper {

    @Mapping(target = "actualDurationMinutes", source = "rental.actualDuration")
    @Mapping(target = "plannedDurationMinutes", source = "rental.plannedDuration")
    @Mapping(target = "rentalEquipments", source = "rental.equipments")
    RentalJpaEntity toEntity(Rental rental);

    @AfterMapping
    default void setRelationships(Rental rental, @MappingTarget RentalJpaEntity result) {
        if (result.getRentalEquipments() != null) {
            result.getRentalEquipments().forEach(equipment -> equipment.setRental(result));
        }
    }
}
```

## Pattern 7 – Abstract Class for Spring Bean Injection

Switch from `interface` to `abstract class` when a mapper needs to **call a Spring-managed service** or **delegate to
another mapper** that cannot be expressed via `uses`. Use `@Autowired` setter injection (the generated subclass does
not support constructor injection for injected fields):

```java

@Mapper(uses = {MoneyMapper.class, DurationMapper.class, DiscountMapper.class, CalculationBreakdownMapper.class})
public abstract class BatchCalculationMapper {

    private MoneyMapper moneyMapper;
    private DiscountDetailMapper discountMapper;

    @Autowired
    public void setMoneyMapper(MoneyMapper moneyMapper) {
        this.moneyMapper = moneyMapper;
    }

    @Autowired
    public void setDiscountMapper(DiscountDetailMapper discountMapper) {
        this.discountMapper = discountMapper;
    }

    @Mapping(target = "plannedDuration", source = "plannedDurationMinutes")
    @Mapping(target = "actualDuration", source = "actualDurationMinutes")
    @Mapping(target = "discount", source = "discountPercent")
    public abstract RentalCostCalculationCommand toCommand(CostCalculationRequest request);

    public CostCalculationResponse toResponse(RentalCostCalculationResult result) {
        return new CostCalculationResponse(
                ...,
        moneyMapper.toBigDecimal(result.totalCost()),
                discountMapper.toResponse(result.discount()),
                ...
        );
    }
}
```

Abstract class pattern for injecting a domain service (e.g. calculator):

```java

@Mapper
public abstract class RentalOverdueMapper {

    protected RentalOverdueCalculator rentalOverdueCalculator;

    @Autowired
    public void setRentalOverdueCalculator(RentalOverdueCalculator rentalOverdueCalculator) {
        this.rentalOverdueCalculator = rentalOverdueCalculator;
    }

    public Integer calculate(Rental rental) {
        return rentalOverdueCalculator.calculateOverdueMinutes(rental);
    }
}
```

## Existing Global Shared Mappers (always prefer reuse)

Before creating a new shared mapper check whether one of these already covers the type:

| Mapper                         | Converts                                                                     |
|--------------------------------|------------------------------------------------------------------------------|
| `shared/mapper/MoneyMapper`    | `BigDecimal` ↔ `Money`                                                       |
| `shared/mapper/UuidMapper`     | `String` ↔ `UUID`                                                            |
| `shared/mapper/InstantMapper`  | `Instant` ↔ `LocalDateTime`                                                  |
| `shared/mapper/DurationMapper` | `Integer` (minutes) ↔ `Duration`                                             |
| `shared/mapper/DiscountMapper` | `BigDecimal`/`Integer` → `DiscountPercent`, `DiscountPercent` → `BigDecimal` |
| `shared/mapper/PageMapper`     | `Pageable` ↔ domain `PageRequest`; Spring `Page<T>` → domain `Page<T>`       |

## Step-by-Step: Adding a New Mapper

1. **Identify the layer** using the placement table above.
2. **Check for existing delegate mappers** needed for field types (VOs, enums, collections).
3. **Choose interface vs. abstract class** — use abstract only if Spring beans must be injected.
4. **Annotate with `@Mapper`** (the global compiler flags handle `componentModel` and `injectionStrategy`).
5. **Add `uses = {...}`** for every delegate mapper whose conversions are needed.
6. **Declare mapping methods** — use `@Mapping` for every non-trivial field:
    - Ignore `createdAt`/`updatedAt` in JPA `toEntity` methods.
    - Use `source = "paramName.field"` for multi-parameter methods.
    - Use `expression = "java(...)"` for derived or conditional values.
    - Use `@Named` + `qualifiedByName` for disambiguating same-type conversions.
7. **Add `@AfterMapping`** only in JPA mappers when bidirectional relationships need wiring.
8. **Build** — `unmappedTargetPolicy=ERROR` will surface any missed fields immediately.

---

## Quick-Reference Checklist

- [ ] Mapper placed in the correct layer package
- [ ] Interface used unless Spring bean injection is required (use abstract class then)
- [ ] `@Mapper(uses = {...})` lists all delegate VO mappers needed
- [ ] JPA `toEntity` methods ignore `createdAt` and `updatedAt`
- [ ] Multi-parameter sources disambiguated with `source = "paramName.field"`
- [ ] `@Named` / `qualifiedByName` used when same type needs different conversions
- [ ] `@AfterMapping` used only in JPA mappers for relationship wiring
- [ ] No Spring `@Component`, `@Service`, or `@Repository` on mapper interfaces/classes (handled by MapStruct)
- [ ] No inline comments in generated/hand-written mapper code (project convention)
