---
description: "Check for dependabot branches, cherry-pick their commits into dependency-upgrade/{today}, run tests, and report or propose a fix"
argument-hint: "[optional: override date, defaults to today YYYY-MM-DD]"
---

You are managing a routine dependency-upgrade consolidation. Follow this workflow exactly, in order. Do not skip
steps and do not push to the remote at any point — pushing requires explicit user permission and is out of scope for
this command unless the user asks for it afterward.

**Input:** $ARGUMENTS (optional date override in `YYYY-MM-DD` format; if empty, use today's date)

---

### Phase 1: Discover dependabot branches

1. Run `git fetch origin --prune` to make sure remote branch refs are current.
2. List remote branches matching dependabot's naming convention:
   `git branch -r --list 'origin/dependabot/*'`
3. **If no branches are found:** report to the user that there are no pending dependabot branches and STOP. Do not
   create a branch.
4. If branches are found, list them to the user (name + last commit subject) before proceeding.

### Phase 2: Create the consolidation branch

1. Determine the target branch name: `dependency-upgrade/{date}`, where `{date}` is the argument override if given,
   otherwise today's date in `YYYY-MM-DD` format (query the actual system date — do not assume).
2. Make sure `origin/main` is current (already fetched in Phase 1).
3. Create the new branch from `origin/main`:
   `git checkout -b dependency-upgrade/{date} origin/main`
   * If a local or remote branch with that exact name already exists, tell the user and ask whether to reuse it or
     pick a different name — do not silently overwrite it.

### Phase 3: Cherry-pick dependabot commits

For each dependabot branch discovered in Phase 1 (process them in a stable order, e.g. alphabetical by branch name):

1. Find the commits unique to that branch relative to `origin/main`:
   `git log --reverse --format=%H origin/main..origin/dependabot/<name>`
2. Cherry-pick each commit in chronological order onto `dependency-upgrade/{date}`:
   `git cherry-pick <sha>`
3. **If a cherry-pick conflicts** (very possible — multiple dependabot branches often touch the same
   `build.gradle`/version-catalog lines): STOP immediately. Do not attempt to auto-resolve. Show the user the
   conflicting files/hunks and ask how they want it resolved (which version to keep, or how to merge the bumps).
   Wait for their answer before continuing the loop.
4. Keep a running list of which commits (branch + SHA + subject) were successfully cherry-picked, for the final
   report.

### Phase 4: Run tests

1. Ensure JDK 21 is used for Gradle (the shell's default `java` may be an older JRE — set `JAVA_HOME` to a JDK 21
   installation before invoking `./gradlew` if needed).
2. Run the full test suite per this repo's convention:
   `./gradlew test "-Dspring.profiles.active=test"`
3. Evaluate the result:
   * **Tests fail:** Investigate the failure output, identify the root cause, and propose a concrete fix (diff or
     description) to the user in chat. Do NOT apply the fix yet. Do NOT commit anything further. STOP and wait for
     the user's explicit instruction on how to proceed.
   * **Tests pass:** proceed to Phase 5.

### Phase 5: Finalize

1. Run `git status` to confirm the working tree is clean (the cherry-picks already created commits; there should be
   nothing left to commit). If Gradle or the test run left stray untracked/modified files that don't belong in the
   branch, leave them alone and mention them — do not force-clean without asking.
2. If there genuinely are uncommitted changes that belong to the upgrade (e.g. a lockfile regenerated by the build),
   stage and commit them with a message following this repo's commit style (see `smart-commit` conventions), e.g.:
   `chore: consolidate dependabot upgrades for {date}`
3. Report to the user:
   * The branch name created.
   * The list of dependabot branches and commits cherry-picked into it.
   * Confirmation that tests passed.
   * That the branch is local-only and has not been pushed — ask if they want it pushed / a PR opened.
