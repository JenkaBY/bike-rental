## Original Request

**As** System Administrator  
**I want** to track all changes made to customer profiles  
**So that** we have a complete audit history for compliance and troubleshooting

## User Story Details

**Description:**  
System must maintain a complete history of all changes made to customer profiles, tracking what changed, when, and by
whom.

**Requirements:**

- Track all field changes (phone, name, email, birthDate, comments)
- Record timestamp of each change
- Store previous and new values
- Immutable audit records (cannot be modified or deleted)
- Query audit history by customer ID

**Acceptance Criteria:**

- All customer profile updates create audit records
- Audit history can be retrieved via API
- Audit records include: timestamp, field name, old value, new value
- System performance is not significantly impacted by audit logging

**Related Requirements:** FR-CL-003 (acceptance criteria: "Сохранение истории изменений")

## Thought Process

This is a technical requirement split from US-CL-003 to keep the core update functionality separate from audit trail
implementation. Key considerations:

1. **Implementation Options:**
    - **Spring Data Envers**: Automatic JPA entity versioning, mature solution, adds database overhead
    - **Custom Event-based Audit**: More control, lighter weight, requires more code
    - **Database Triggers**: DB-level solution, technology-agnostic but harder to maintain

2. **Recommended Approach**: Spring Data Envers
    - Already available in Spring Boot ecosystem
    - Automatic revision tracking with minimal code
    - Built-in query API for historical data
    - Standard solution with community support

3. **Data Model:**
    - Envers creates `customers_aud` table automatically
    - Revision info table tracks when/who made changes
    - Join queries retrieve historical versions

4. **API Design:**
    - GET `/api/customers/{id}/history` - retrieve all changes
    - GET `/api/customers/{id}/history/{revisionId}` - specific version
    - Response includes revision metadata + customer state

```
        ├── CustomerAuditHistoryResponse
        └── CustomerRevisionDto
```

**Dependencies:**

```gradle
implementation 'org.springframework.boot:spring-boot-starter-data-envers'
```

**API Endpoints:**

- `GET /api/customers/{id}/history` - All revisions for customer
    - Response: List of revisions with metadata
- `GET /api/customers/{id}/history/{revisionId}` - Specific revision
    - Response: Customer state at that revision

**Envers Configuration:**

```java
spring.jpa.properties.org.hibernate.envers.audit_table_suffix=_aud
spring.jpa.properties.org.hibernate.envers.revision_field_name=rev
spring.jpa.properties.org.hibernate.envers.revision_type_field_name=revtype
```

**Database Schema (Auto-generated by Envers):**

- `customers_aud` - Audit table mirroring customers structure
- `revinfo` - Revision metadata (timestamp, revision number)
